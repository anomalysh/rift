title: Design notes
description: The decisions that are easy to get wrong, and why rift makes them the way it does.
---

These are the parts of rift that are easy to get wrong, kept here so the
reasoning is not lost. Each is a deliberate choice with a failure it avoids.

## Subdomain claims are leases, not locks

A tunnel row in Postgres is a claim on a subdomain. A gateway that crashed
without releasing its rows would hold those subdomains forever, so a **reaper**
collects rows whose agents stopped heartbeating. On boot, riftd also clears the
rows its own previous run left behind, returning those subdomains immediately
instead of waiting out a heartbeat timeout.

## Heartbeats are application-level

WebSocket ping/pong frames are answered by intermediaries and prove only that a
proxy between the agent and the gateway is alive. rift's heartbeat is an
application-level `ping`/`pong` control message instead, so it proves the agent's
own event loop is running. A gateway that hears no heartbeat within its timeout
closes the connection and reaps the tunnel.

## Reconnect takes over before counting

An agent whose socket dropped still owns its tunnel row until the reaper runs. If
a reconnect counted that row against the per-token tunnel limit, a token with
`max_tunnels=1` could never reconnect to its own subdomain — the old row would
occupy its only slot. So a reconnect reclaims its own row first, then counts. A
`hello` for a subdomain owned by the same token is treated as a takeover: the
older connection is sent `shutdown{reason:"replaced"}` and closed.

## One writer per socket

A WebSocket permits exactly one concurrent writer. Every frame — including the
final shutdown frame — therefore goes through a single writer goroutine, so
concurrent streams multiplexed over one connection can never interleave a
half-written frame.

## Routing stops when close is requested, not when the socket finishes closing

A well-behaved agent stops reading the moment it receives a shutdown frame, so it
never sends a close frame back, and the graceful WebSocket close handshake
stalls. Waiting for that handshake to complete would leave the subdomain
answering `502` instead of `404` and unable to be reclaimed. So routing for a
tunnel stops when close is *requested*, not when the socket has finished closing.

## Two implementations of one protocol drift

The protocol has two implementations — `tunnelproto` in Go and `protocol.ts` in
TypeScript — and two implementations of one wire format inevitably drift.
`cli/test/conformance.test.ts` asserts the TypeScript frames byte-for-byte
against fixtures generated by the Go encoder. That test is what keeps the two
honest; a change to one side that is not mirrored in the other fails it.

## No HSTS on the tunnel wildcard

Each tunnel is an untrusted, user-controlled subdomain. An HSTS header served
from one malicious tunnel — especially with `includeSubDomains` — would be cached
by the browser against the **parent** domain and could brick unrelated sibling
subdomains. So the wildcard site block sends no `Strict-Transport-Security`
header at all; HSTS belongs only on hostnames the operator fully controls.

## The admin API is never exposed

riftd's admin listener mints tokens and manages reservations. It is deliberately
not proxied by any Caddy site block, and its port is never published on the host.
Exposing it publicly would put token minting one weak credential away from a full
takeover, so it is reachable only from the host itself or over an SSH tunnel.
